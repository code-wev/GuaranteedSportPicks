BADB FRONTEND DOCUMENTATION
Professional Technical Project Document

Document Purpose
This document provides a complete technical and architectural overview of the BADB frontend application. It is written in a formal documentation style so it can be pasted directly into Google Docs or reused as a polished project handover document.

System Name
BADB Final Version Frontend

Primary Responsibility
The frontend delivers the public website, customer account experience, subscription and pick purchasing interfaces, user dashboard, and full administrative dashboard for platform operations.


==================================================
1. EXECUTIVE SUMMARY
==================================================

The BADB frontend is a Next.js App Router application built with React and Tailwind CSS. It communicates with the backend through Redux Toolkit Query and is organized into public pages, authenticated dashboard pages, and admin operations pages.

The application supports three major interaction layers:

1. Public website experience
This includes the home page, packages, free picks, blog pages, about page, and contact page.

2. Authenticated user experience
This includes login, signup, email verification, password recovery, profile management, subscription management, pick access, purchase flows, and the user analytics dashboard.

3. Admin operations experience
This includes management interfaces for users, orders, picks, newsletters, blogs, testimonials, and affiliate workflows.

The frontend is built to operate closely with the BADB backend and depends on backend-controlled billing status, access state, and analytics summaries.

==================================================

The frontend follows a layered application structure.

Primary architectural layers
- app routes
- reusable UI components
- API integration slices
- Redux provider and store
- static assets

Main directory structure

frontend/
  app/
  components/
  feature/
  provider/
  store/
  utils/

Design characteristics
- public pages are built from reusable content components
- dashboards are composed from route-specific page files and reusable dashboard widgets
- API communication is centralized into RTK Query slices
- auth state is primarily represented through a cookie-stored JWT token
- backend state is considered authoritative for purchases, subscriptions, and access control

7. API BASE URL AND AUTH HEADER STRATEGY
==================================================

Configuration file
frontend/utils/utils.js

Current API base URL
http://localhost:5000/api/v1

Authentication strategy
Most API slices read the JWT token from browser cookies and attach it to the Authorization header as:

Authorization: Bearer <token>

Important note
The frontend does not treat client navigation as proof of access. It still depends on backend authorization for all protected operations.

- /reset-password

Protected route roots
- /dashboard
- /profile
- /settings

- unauthenticated users are redirected to login when visiting protected areas

Important implementation note
This middleware protects authenticated entry, but role-level restrictions are still enforced by the backend.

15. KEY USER JOURNEYS
==================================================

15.1 Signup and Email Verification Journey

1. User opens the signup page.
2. Frontend submits registration data using useRegisterMutation.
3. Backend creates the account and sends a verification email.
4. User opens the verification link.
5. Frontend submits the verification token using useVerifyEmailMutation.
6. User becomes eligible to log in.


15.2 Login Journey

1. User submits email and password on the login page.
2. Frontend sends credentials using useLoginMutation.
3. Token is stored in cookie.
4. Middleware permits access to protected routes.
5. Dashboard pages begin using authenticated API hooks.


15.3 Password Recovery Journey

1. User requests a reset from the forgot-password page.
2. Frontend triggers useForgetPasswordMutation.
3. User receives reset link.
4. Reset page submits email, token, and new password using useResetPasswordMutation.
5. User returns to login flow.


15.4 Subscription Purchase Journey

Primary page
frontend/app/dashboard/purchase/page.js

Flow
1. User selects a package.
2. User selects one or more sports.
3. If seasonal package is selected, user enters days and price.
4. Frontend validates the form locally.
5. Frontend creates a subscription using useCreateSubscriptionMutation.
6. Backend returns a Stripe payment link.
7. Browser redirects to Stripe checkout.
8. After payment, the user returns to the frontend.
9. Pages refetch backend state to show updated subscription access.


15.5 Individual Pick Purchase Journey

Primary page
frontend/app/dashboard/purchase/page.js

Flow
1. Frontend loads premium active picks from the board.
2. User chooses either prepaid purchase or pay-after-win.
3. Frontend creates a pick purchase using useCreatePickPurchaseMutation.
4. Backend returns Stripe payment link.
5. Browser redirects to Stripe.
6. Purchase status later appears through my-picks and purchase queries.


15.6 My Picks Journey

Primary page
frontend/app/dashboard/my-picks/page.js

Data sources
- getMyPickPurchases
- getMyAccessiblePicks
- getMySubscription

Page behavior
- divides visible content into Active and Completed tabs
- renders purchase badges for prepaid or pay-after-win purchases
- displays subscription access badge when all premium picks are unlocked
- shows expert writeup for accessible picks


15.7 Profile Management Journey

==================================================
24. IMPORTANT IMPLEMENTATION NOTES
==================================================

1. Token dependency
Authenticated API calls depend on token availability in cookies.

2. Billing truth source
The frontend should always refresh backend state after Stripe redirect. Redirect completion alone does not guarantee access.

3. Naming consistency
The newslatter spelling is intentionally preserved to match backend routes.

4. Role security
Admin pages exist in the route tree, but backend role validation remains the final source of access control.

5. Backend-driven analytics
Most dashboard analytics are server-aggregated, which keeps the frontend simpler and more presentation-focused.
