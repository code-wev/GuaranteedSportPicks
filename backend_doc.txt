BADB BACKEND DOCUMENTATION
Professional Technical Project Document

Document Purpose
This document provides a complete technical reference for the BADB backend application. It is written as a project handover and engineering reference document so it can be pasted directly into Google Docs or used as a formal internal project document.

System Name
BADB Final Version Backend

Primary Responsibility
The backend serves as the main business logic and data-processing layer for the sports picks platform. It manages users, authentication, premium picks, subscription billing, single-pick purchases, blog content, affiliate operations, newsletter delivery, testimonials, and dashboard reporting.


==================================================
1. EXECUTIVE SUMMARY
==================================================

The BADB backend is a modular Express and TypeScript API built on top of MongoDB. The application follows a feature-based module structure and separates responsibilities across route handlers, controllers, services, models, and utility helpers.

The platform supports three major operational areas:

1. Public platform services
These include article retrieval, public pick board access, testimonials, and public-facing onboarding flows such as signup and password recovery.

2. Authenticated user services
These include account management, subscription purchase and cancellation, premium pick access, pick purchase history, dashboard analytics, and newsletter preferences.

3. Administrative services
These include user management, pick creation and settlement, order reporting, affiliate management, newsletter campaign delivery, testimonial moderation, and article publishing.

The backend also integrates with Stripe for billing and uses webhook-driven payment confirmation to activate subscriptions and finalize pick-purchase states.

==================================================
6. AUTHENTICATION AND AUTHORIZATION MODEL
==================================================

Authentication method
The system uses JWT-based authentication. After a successful login, the frontend stores the token and sends it back in the Authorization header as a bearer token.

Authorization model
The platform supports two roles:
- ADMIN
- USER

Core middleware used
- isAuthorized
- authorizedRoles
- optionalUser

Behavior summary
- isAuthorized validates the JWT and attaches the user to the request
- authorizedRoles restricts access to selected roles
- optionalUser allows the request to continue even if no user token is present, but still enriches the request when a token exists

9. CORE BUSINESS FLOWS
==================================================

9.1 User Registration and Verification

Registration flow
1. User submits signup information.
2. Backend validates email, phone number, and password rules.
3. Backend checks duplicate email and duplicate phone number.
4. Password is hashed.
5. Email verification token is generated.
6. User record is saved with unverified email state.
7. Verification email is sent to the user.

Verification flow
1. User opens verification link.
2. Frontend sends email and token to the verification API.
3. Backend validates the token and expiry.
4. User account is marked verified.


9.2 Login Flow

1. User submits email and password.
2. Backend verifies the user exists.
3. Backend confirms email has already been verified.
4. Password is compared using bcrypt.
5. JWT token is generated and returned.


9.3 Password Recovery Flow

Forget password
1. User submits email.
2. Backend generates reset token and expiry.
3. Reset email is sent.

Reset password
1. User submits email, token, and new password.
2. Backend validates token and expiry.
3. Backend checks the new password is different from the existing password.
4. Password is hashed and updated.


9.4 Premium Pick Access Flow

Premium access is determined by:
- active paid subscription
- successful individual purchase of the pick
- or the pick being free

For users without access, the backend sanitizes premium picks by hiding:
- selected_team
- units
- writeup

This allows the pick board to show that a premium pick exists without leaking the premium content itself.


9.5 Subscription Billing Flow

Recurring plans
- DAILY
- WEEKLY
- MONTHLY

Seasonal plan
- SEASONAL

Flow
1. Frontend requests subscription creation.
2. Backend creates a pending subscription record.
3. Backend creates a Stripe checkout session.
4. User is redirected to Stripe.
5. Stripe webhook confirms payment.
6. Backend activates the subscription and updates billing dates.


9.6 Single Pick Purchase Flow

Supported payment modes
- PREPAID
- PAY_AFTER_WIN

PREPAID behavior
- payment is captured as part of normal checkout completion

PAY_AFTER_WIN behavior
- card is authorized first
- payment is captured only if the pick result becomes win
- authorization is cancelled if result becomes loss or void


9.7 Affiliate Revenue Flow

1. User signs up using a referral code.
2. Backend resolves the code to an approved affiliate.
3. User is linked to that affiliate.
4. If the user buys or renews a subscription, a commission entry may be created.
5. Approved affiliates can request withdrawals.
6. Admin can update withdrawal statuses.


==================================================
10. STRIPE WEBHOOK BEHAVIOR
==================================================

Webhook endpoint
/api/v1/subscription/webhook

Important handled events

checkout.session.completed
- activates seasonal subscriptions
- activates recurring subscriptions
- marks prepaid pick purchases as paid
- marks pay-after-win purchases as authorized

invoice.paid
- updates recurring subscription billing period
- records renewal affiliate commission when appropriate

invoice.payment_failed
- marks subscription as failed

customer.subscription.deleted
- marks subscription as cancelled

checkout.session.expired
- cancels abandoned pending pick purchases

Important operational note
Stripe webhook processing is the real source of truth for final payment state. Frontend redirects alone should never be treated as completed billing confirmation.

12. VALIDATION RULES BY FEATURE
==================================================

Auth validation highlights
- registration enforces strong password format
- email fields must be valid
- password reset requires password confirmation
- password change requires new password confirmation

Pick validation highlights
- pick purchase requires valid Mongo pick ID
- payment model must be PREPAID or PAY_AFTER_WIN
- market_type must be moneyline, spread, or totals
- confidence must be low, medium, or high

Subscription validation highlights
- at least one sport must be selected
- seasonal plans require seasonalDays and seasonalPrice
- recurring plans require price
- compatibility fields such as customDays and customPrice are still supported

User validation highlights
- self-delete requires exact phrase DELETE MY ACCOUNT
- profile update requests must contain at least one field

Affiliate validation highlights
- withdrawal amount must be positive
- withdrawal update statuses are controlled
- affiliate approval only allows approved or declined

Article validation highlights
- title minimum 5 characters
- content minimum 20 characters
- category required

Testimonial validation highlights
- review minimum 20 characters
- rating must be between 1 and 5

Newsletter validation highlights
- campaign title, subject, and content are required for sending newsletters

16. RECOMMENDED INTEGRATION CHECKLIST
==================================================

For any frontend or external client integrating with this backend:

1. Use /auth/login to obtain a JWT token.
2. Send the token as a bearer token in the Authorization header.
3. Use /user/profile/me to initialize the authenticated session.
4. Use /subscription/my-active to determine premium subscription coverage.
5. Use /picks/my-access to determine which picks are fully visible to the user.
6. Redirect the user to Stripe using the paymentLink returned by purchase endpoints.
7. After returning from Stripe, refetch API state instead of trusting the browser redirect alone.
